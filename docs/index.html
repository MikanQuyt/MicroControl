◈ SYSTEM OVERVIEW

How the car works

The FPV car is a two-MCU system. The ESP32-CAM serves a web UI and streams video over WiFi. When you press a button, it sends a 1-byte SPI command to the ATmega128, which handles all real-time hardware control.

📡 ESP32-CAM (Master)

  • Hosts WiFi access point connection
  • Serves web control page (port 80)
  • Streams OV2640 camera video (port 81)
  • Sends 1-byte commands via SPI to Mega128

🎛️ ATmega128 (Slave)

  • Receives SPI commands, drives motors via TB6612FNG
  • Reads MPU6050 gyro for heading correction
  • Counts H206 encoder pulses for precise turns
  • Measures distances with 3× HC-SR05 ultrasonic
  • Runs autonomous obstacle avoidance (Auto mode)

01 SPI COMMUNICATION

SPI Slave Receiver

The ATmega128 runs as an SPI slave. When the ESP32-CAM pulls SS low and clocks out a byte, the SPI interrupt fires and stores the command.

⚡ SPI_SlaveInit() + ISR ▶ 
C
void SPI_SlaveInit(void) {
    // SS(PB0), SCK(PB1), MOSI(PB2) = Input
    DDRB &= ~((1 << PB0) | (1 << PB1) | (1 << PB2));
    // MISO(PB3) = Output
    DDRB |= (1 << PB3);
    // Enable SPI + Interrupt, Slave mode
    SPCR = (1 << SPE) | (1 << SPIE);
}

ISR(SPI_STC_vect) {
    received_cmd = SPDR;   // Read the byte from SPI data register
    new_cmd_flag = 1;      // Set flag for main loop to process
}

Key point: The ISR only stores the byte and sets a flag. All command processing happens in the main() loop to avoid blocking inside the ISR.

02 I2C (TWI) DRIVER

Hardware I2C Bus

Uses the ATmega128's built-in TWI hardware on PC0 (SCL) and PC1 (SDA). Clock speed = 100 kHz.

🔌 I2C Functions ▶
SCL frequency = F_CPU ÷ (16 + 2 × TWBR × Prescaler) = 16 MHz ÷ (16 + 2 × 72 × 1) = 100 kHz

I2C_Init()

Sets TWBR=72, prescaler=1, enables the TWI module. This gives a 100kHz I2C clock — the standard speed for MPU6050.

I2C_Start() / Stop()

Start: Sets TWSTA + TWINT + TWEN, waits for TWINT flag. Stop: Sets TWSTO, adds 10µs delay for bus release.

I2C_Write(data)

Loads data into TWDR, triggers transmission, waits for completion.

I2C_ReadACK() / ReadNACK()

ACK: Read byte and acknowledge (more bytes expected). NACK: Read last byte and signal end of read.

03 MPU6050 IMU

Gyroscope Z-axis

The MPU6050 provides angular velocity around the Z-axis (yaw). This is used for heading correction when driving forward and to verify turn accuracy.

🧭 MPU6050 Init & Read ▶ 
C
void MPU6050_Init(void) {
    _delay_ms(100);                                 // Wait for power-up
    MPU6050_WriteReg(MPU6050_PWR_MGMT_1, 0x00);   // Clear SLEEP bit
    _delay_ms(50);
    MPU6050_WriteReg(MPU6050_GYRO_CONFIG, 0x00);  // ±250°/s range
}

int16_t MPU6050_ReadGyroZ(void) {
    uint8_t high = MPU6050_ReadReg(0x47);  // GYRO_ZOUT_H
    uint8_t low  = MPU6050_ReadReg(0x48);  // GYRO_ZOUT_L
    return (int16_t)((high << 8) | low);
}

Sensitivity: At ±250°/s range, the gyro outputs 131 LSB per °/s. So a raw reading of 655 means the car is rotating at ~5°/s.

04 H206 WHEEL ENCODER

ADC Pulse Counting

The H206 is a slotted disk encoder that outputs an analog signal. As the wheel spins, the signal alternates high/low through the slots. We read the ADC and detect rising edges.

📊 ADC Init & Encoder Update ▶ 
C
void Encoder_Update(void) {
    // Left encoder on ADC0
    uint16_t adc_left = ADC_Read(0);
    uint8_t left_now = (adc_left > ENCODER_THRESHOLD) ? 1 : 0;
    if (left_now && !enc_left_prev) {  // Rising edge detected!
        enc_left_count++;               // Increment pulse counter
    }
    enc_left_prev = left_now;
    // ... same for right encoder on ADC1
}

How it works: The ADC reads the analog voltage. If it's above 512 (2.5V midpoint), it's "HIGH". We only count transitions from LOW→HIGH (rising edge), giving us 20 pulses per wheel revolution.

⚠️ Important: Encoder_Update() must be called frequently (every main loop iteration) for accurate counting. Missing calls = missed pulses.

05 MOTOR CONTROL

TB6612FNG Driver

The TB6612FNG is a dual H-bridge motor driver. Each motor has two direction pins (AIN1/AIN2 or BIN1/BIN2) and one PWM speed pin.

⚙️ Motor Direction Logic ▶
ActionAIN1 (PC2)AIN2 (PC3)BIN1 (PC4)BIN2 (PC5)
Forward0101
Backward1010
Turn Left1 (rev)001 (fwd)
Turn Right01 (fwd)1 (rev)0
Stop0000

Speed compensation: Motor A (left) runs at 70% and Motor B (right) at 80% of the set PWM to balance uneven motors and make the car drive straight.

OCR3A = current_speed × MOTOR_A_PERCENT ÷ 100
OCR3B = current_speed × MOTOR_B_PERCENT ÷ 100
🔧 PWM Configuration (Timer3) ▶ 
C
void PWM_Init(void) {
    DDRE |= (1 << PE3) | (1 << PE4);          // OC3A, OC3B as output
    TCCR3A = (1<<COM3A1) | (1<<COM3B1) | (1<<WGM30); // Fast PWM 8-bit
    TCCR3B = (1<<WGM32) | (1<<CS31) | (1<<CS30);   // Prescaler=64
}
PWM freq = 16 MHz ÷ (64 × 256) ≈ 976 Hz
06 PRECISE TURNING

Encoder-Based Angle Control

Instead of turning for a fixed time (inaccurate), the car counts encoder pulses to achieve an exact turn angle.

📐 The Math

arc = (angle ÷ 360) × π × wheel_spacing
pulses = (arc ÷ wheel_circumference) × slots_per_rev

Example for 90° turn:

arc = (90 ÷ 360) × 3.14 × 15 cm = 11.78 cm
pulses = (11.78 ÷ 21) × 20 = 11 pulses
🔄 Motor_TurnLeftAngle() / Motor_TurnRightAngle() ▶ 
C
void Motor_TurnLeftAngle(uint16_t angle_deg) {
    uint16_t target = Pulses_For_Angle(angle_deg);
    Encoder_Reset();       // Zero both counters
    Motor_TurnLeft();      // Start turning
    while (enc_right_count < target) {  // Count RIGHT wheel
        Encoder_Update();   // Keep polling ADC
    }
    Motor_Stop();          // Exact angle reached!
}

Why count the opposite wheel? During a left turn, the left wheel goes backward and the right wheel goes forward. We count the forward-moving wheel because its encoder gives more consistent readings.

07 HEADING CORRECTION

Gyro-Based Straight Driving

Even with speed compensation, wheels may drift. The gyro provides real-time feedback to adjust individual motor speeds and keep the car driving straight.

🎯 Motor_ForwardStraight() Algorithm ▶

How correction works:

  • Gyro Z > +50 → Car drifting right → Boost left motor, slow right motor
  • Gyro Z < -50 → Car drifting left → Boost right motor, slow left motor
  • |Gyro Z| ≤ 50 → Within deadzone → No correction (noise)
correction = (gz − DEADZONE) × HEADING_GAIN ÷ 131
PWM_A = base_A + correction   |   PWM_B = base_B − correction

The loop runs every 10ms for the specified duration, giving smooth real-time corrections.

08 ULTRASONIC SENSORS

HC-SR05 Distance Measurement

Three ultrasonic sensors (front, left, right) detect obstacles by measuring the time for a sound pulse to bounce back.

This comes from: speed of sound = 343 m/s = 0.034 cm/µs. Round trip → divide by 2. So: distance = time × 0.034 ÷ 2 = time ÷ 58.8 ≈ time ÷ 58.

Timeout protection: 60,000 iterations waiting for echo start, 25,000µs max echo duration (~4.25m range). Returns 0 on timeout.

09 BELL / BUZZER

Connection Indicator

A simple buzzer on PB7. Rings for 200ms when the car receives the 'C' (Connect) command.

Bell_Init() → PB7 = output, LOW   |   Bell_Ring() → PB7 HIGH, wait 200ms, PB7 LOW

10 AUTO DRIVE

Obstacle Avoidance

When in Auto mode (is_active == 2), the car continuously reads all 3 sensors and makes decisions.

🤖 Auto_Drive() Decision Tree ▶
PriorityConditionAction
1 (highest)Front clearMotor_ForwardStraight(200) — forward 200ms with gyro correction
2Front blocked, Left clearMotor_TurnLeftAngle(90) — precise 90° left turn
3Front+Left blocked, Right clearMotor_TurnRightAngle(90) — precise 90° right turn
4 (lowest)All directions blockedReverse 400ms → Stop → 90° left turn

Blocked = distance < 6cm OR sensor returned 0 (error/timeout). The 50ms delay between cycles prevents sensor interference.

11 MAIN PROGRAM

Initialization & Main Loop

The entry point initializes all hardware modules, then runs an infinite loop that updates encoders, processes SPI commands, and runs auto mode.
